## 1. 🗂️ 信息架构 (7.5/10)

### 1.1 导航模型

```
┌─────────────────────────────────────────────────┐
│  modules (模组选择)                              │
│    └→ lobby (大厅)                               │
│         ├→ character (调查员管理)                 │
│         │    └→ creator (角色创建)                │
│         ├→ settings (AI引擎设置)                  │
│         ├→ saves (存档管理)                       │
│         └→ devlog (开发者日志) → 独立视图          │
│              └→ story (剧情主界面) ★核心视图       │
│                   ├─ [9面板] chat/char/inv/       │
│                   │   storage/journal/npc/        │
│                   │   growth/clues/combat         │
│                   ├─ [Modal] map / dice / save    │
│                   └─ [持久] toast / confirm       │
└─────────────────────────────────────────────────┘
```

### 1.2 优点

- ✅ **清晰的三层导航**: 模组选择 → 大厅 → 剧情，层级明确
- ✅ **9 面板 tab 切换**: story_view 使用 `v-if` 懒加载模式，首次点击才渲染面板，性能良好
- ✅ **一致的返回路径**: 所有子屏都有"返回"按钮，不会迷路
- ✅ **Bootstrap 原生控件**: 利用 Bootstrap 的 tab/nav 组件，用户认知负担低
- ✅ **模组隔离**: 每个模组有独立存档，适合多战役管理

### 1.3 问题

| # | 问题 | 严重度 | 建议 |

| IA-01 | **9个 tab 缺少分组/优先级** | 中 | 将 chat/combat 提升为主要 tab，其余折叠为"更多"下拉菜单。当前全部平铺，移动端 tab 栏溢出需横向滚动 |

## 4. 🔄 反馈循环 (8.0/10)

### 4.1 操作→反馈分析

| 操作 | 反馈机制 | 延迟 | 评分 |
|------|----------|------|------|
| 发送消息 | 输入框禁用 + "守秘人正在推演中..." + `isLoading` 状态 | <100ms 视觉响应 | 🟢 |
| AI 返回结果 | 消息追加到聊天 + 自动滚动到底部 | 取决于 API | 🟢 |
| 技能检定 | `dice-pulse` 动画 (350ms) + 骰面显示 + 结果着色 | 即时 | 🟢 |
| 存档成功 | Toast 通知 (`coc-toast-success`) + 聊天系统消息 | <50ms | 🟢 |
| 战斗命中 | `combat-hit-flash` 动画 (760ms) | 即时 | 🟢 |
| SAN 下降 | 多层次反馈: overlay 暗化 → `san-shock` 滤镜 → `san-shake` 震动 → `madness-border` 闪烁 | 即时 | 🟢 |
| 物品 tooltip | `tooltipIn` 动画 (150ms ease-out) | 即时 | 🟢 |
| 确认对话框 | `modal-overlay` + 聚焦按钮 | 即时 | 🟢 |

### 4.2 优点

- ✅ **13 个 keyframe 动画**覆盖核心交互：掷骰、战斗、SAN、物品 rarity 光效
- ✅ **多层次 SAN 反馈系统**：这是该项目的亮点 — 从轻微的边缘暗化(low-san) 到疯狂的视觉扭曲(san-madness) 和死亡脉冲(san-death-pulse)
- ✅ **战斗动画完整**：命中(hit-flash)、暴击(crit-flash)、闪避(dodge-flash)、格挡(block-flash)、击杀(kill-flash)
- ✅ **Toast 通知系统**：4种类型 (success/warning/danger/info) + 自动消失 + 最高5条限制
- ✅ **确认对话框**：支持 danger 模式，防止误删存档/角色
- ✅ **loading 状态管理**：`isLoading` 全局锁，防止重复提交；`isSyntheticMoveInFlight` 防止移动连发

### 4.3 问题

| # | 问题 | 严重度 | 建议 |
|---|------|--------|------|
| FB-01 | **AI 响应期间缺少进度指示** | 中 | "守秘人正在推演中..." 是静态文字，无法区分"正在等待网络"和"AI 正在生成"。建议加省略号动画 (`...`) 或进度条 |
| FB-02 | **长内容 AI 响应无分段加载** | 低 | 目前一次性渲染整个 AI 响应。若响应极长，用户需等待全部完成才能阅读 |
| FB-03 | **聊天区新消息无未读提示** | 低 | 用户滚动查看历史时，新消息不会触发"新消息"提示，可能遗漏 |

## 5. ⚠️ 错误处理 (7.5/10)

### 5.1 覆盖场景

| 场景 | 处理方式 | 评分 |
|------|----------|------|
| JS 全局错误 | `window.onerror` → 致命崩溃全屏遮罩 (红色) | 🟢 |
| Promise 崩溃 | `unhandledrejection` → 同上 | 🟢 |
| Vue 渲染错误 | `console.error` 劫持 → 同上 | 🟢 |
| 组件挂载失败 | `console.warn` 劫持 → 同上 | 🟢 |
| AI 请求超时 | AbortController 30s 超时 + 最多3次重试 + 指数退避 | 🟢 |
| AI Worker 崩溃 | 自动回退到同步 fetch，解决所有 pending | 🟢 |
| localStorage 配额超限 | `QuotaExceededError` 检测 → Toast 提示 + 聊天系统通知 | 🟢 |
| 存档损坏 | loadGame 返回 false → Toast "存档损坏或不存在" | 🟢 |
| 导入失败 | try-catch → Toast "导入失败：{错误}" | 🟢 |
| 无效骰子表示法 | 自定义骰子输入验证 → Toast "无效的骰子表示法" | 🟢 |
| 网络断开 | 无专门处理 | 🟠 |
| API Key 错误 | HTTP 状态传递给 `formatAiError` → 系统消息 | 🟡 |

### 5.2 优点

- ✅ **4层错误捕获装甲**: 全局 error → unhandledrejection → console.error 劫持 → console.warn 劫持
- ✅ **致命崩溃屏 XSS 安全**: 使用 `escapeHtml()` 转义所有用户数据后再 innerHTML
- ✅ **AI 请求指数退避重试**: `[0, 800, 1600]ms`，支持环境变量覆盖
- ✅ **Worker 优雅降级**: Worker 不可用时自动 fallback 到同步 fetch
- ✅ **存储配额主动检测**: 显示 `usedBytes/quotaBytes` 比例，超80% 告警

### 5.3 问题

| # | 问题 | 严重度 | 建议 |
|---|------|--------|------|
| EH-01 | **无 `navigator.onLine` 检测** | 🟡 中 | 网络断开时 AI 请求会静默失败，用户看到超时错误但不知道是断网。建议监听 `online/offline` 事件并 Toast 提示 |
| EH-02 | **致命崩溃屏只有"刷新重试"一个选项** | 低 | 可增加"清除缓存并重试"和"导出存档"按钮 |
| EH-03 | **API 401/403 错误无区分提示** | 低 | 目前所有 HTTP 错误统一走 `formatAiError`，但密钥错误(401)和权限错误(403)应有不同指引（如"请检查 API Key"）|
| EH-04 | **找不到组件时直接 throw Error 阻断启动** | 中 | `app.js` 中找不到 ViewLobby/ViewCreator/ViewStory 直接 throw，虽然被全局错误拦截，但页面完全不可用。建议改为降级显示"组件加载失败"占位 |

## 6. 📱 移动端 (6.5/10)

### 6.1 已实现

- ✅ **viewport 配置**: `width=device-width, initial-scale=1.0, maximum-scale=1.0`
- ✅ **touch-action: manipulation**: body + 所有按钮/链接，禁用双击缩放和长按菜单
- ✅ **safe-area 适配**: `#app` 使用 `env(safe-area-inset-bottom/left/right)`
- ✅ **apple-touch-icon**: 已配置
- ✅ **@media (max-width: 768px)** 断点:
  - 面板 max-height 从全屏缩至 60-70vh
  - tab 按钮缩小 padding
  - 工具栏 flex-wrap
  - 聊天框高度从 60vh 降至 45vh
- ✅ **PWA manifest + Service Worker**: 离线可用
- ✅ **触屏长按加点**: `@touchstart.prevent` + `@touchend` 实现属性加点连按

### 6.2 问题

| # | 问题 | 严重度 | 建议 |
|---|------|--------|------|
| MB-01 | **9个 tab 在 375px 宽度必然溢出** | 🔴 高 | 移动端 tab 栏需要横向滚动才能看到全部标签。建议改为汉堡菜单或底部导航 |
| MB-02 | **移动端触摸目标偏小** | 🔴 高 | WCAG 建议 44×44px。大量按钮使用 `btn-sm` (padding: 4px 6px, font-size: 0.75rem)，实际触摸区域约 28-32px。故事工具栏的小图标按钮 (⬅️, ⚙️, 🎲, 💾) 更小 |
| MB-03 | **modal 在移动端占比过大** | 🟡 中 | `.modal-content-custom { width: 95%; max-height: 85vh }` — 95%宽度下 padding 20px 后内容区非常窄 |
| MB-04 | **只有一个媒体查询断点** | 🟡 中 | 仅 `@media (max-width: 768px)`，缺少平板 (768-1024px) 和大屏 (>1200px) 优化 |
| MB-05 | **`maximum-scale=1.0` 阻止用户缩放** | 低 | 可访问性考虑：低视力用户可能需要放大页面。建议改为 `maximum-scale=2.0` 或去掉限制 |
| MB-06 | **safe-area 仅用于 #app** | 低 | 固定定位的 toast (.coc-toast-layer) 和 footer 版本号没有 safe-area 适配，可能被 iPhone 底部横条遮挡 |
