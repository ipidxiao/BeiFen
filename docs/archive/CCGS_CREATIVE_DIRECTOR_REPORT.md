## 一、🎯 产品愿景 (Product Vision) — 8/10

### 1.1 核心价值主张

CoC 7th Engine 的定位一句话：**「AI 当守秘人，你当调查员——零门槛、纯前端、全中文的克苏鲁TRPG跑团引擎」**。

不是"跑团辅助工具"，而是"AI驱动的TRPG完整解决方案"。这个定位在中文TRPG市场具有明确的价值锚：

1. **解决「缺KP」问题**: 中文CoC圈子最大痛点是会带团的守秘人稀缺。AI守秘人让任何人随时开始游戏。
2. **规则自动化**: 骰子检定/战斗/理智/重伤/神话典籍/法术施放全部由引擎处理——玩家只需扮演角色。
3. **连续叙事**: "零停顿执行"（文档原话），AI连续推动剧情不经人工分阶段介入——这是与聊天bot的本质区别。

### 1.2 目标用户画像

| 用户层 | 规模估计 | 痛点 | 本产品解法 |
|--------|:--------:|------|-----------|
| 🥇 CoC新手玩家 | 大 | 想玩但找不到KP/不想社交 | AI守秘人零门槛开团 |
| 🥈 独狼资深玩家 | 中 | 想跑单人团，没人带 | 全自动单人CoC战役 |
| 🥉 KP辅助工具 | 中 | 想自动化战斗/检定等枯燥部分 | 引擎可配合人类KP使用 |
| 🏅 CoC规则学习者 | 小 | 想了解CoC 7e机制 | 引擎的规则透明性作为教学工具 |

### 1.3 愿景缺陷

| 问题 | 严重度 | 说明 |
|------|:------:|------|
| 缺乏商业模式定义 | 🟡 | 是开源社区项目还是商业产品？API key由用户自行配置说明暂未考虑变现 |
| AI依赖单一供应商 | 🟡 | 当前硬编码 `api.deepseek.com`，供应商风险集中 |
| 多人联机缺失 | 🟡 | 目标用户含"不想社交的玩家"，但未规划远程多人模式 |
| 移动端体验未知 | 🟡 | 声明"纯前端SPA"但未见移动端适配测试报告 |

> **创意总监注**: 产品愿景足够锋利——"AI守秘人"这五个字就是最好的elevator pitch。但需要明确：这是工具还是平台？免费还是付费？单机还是联网？这些战略选择题必须在上线前回答。

### 3.4 克苏鲁性的缺失

| 缺失项 | 说明 | 影响 |
|--------|------|------|
| 无BGM/环境音效 | `js/audio/sfx.js` 文件存在但功能极其有限。无心跳声/低语/风声等克系音频资产 | 氛围沉浸的听觉维度空白 |
| 典籍阅读缺少仪式感 | 目前是函数调用返回文本结果，缺少"阅读禁忌知识"的UI仪式感（如逐页翻动效果、逐渐扭曲的文字） | 降低神话典籍的恐怖冲击力 |
| SAN效果仅视觉 | 缺少叙事层面的SAN反馈——如AI叙事文本随SAN降低逐渐变得偏执/幻觉化 | AI铁律中有此意图，但narrativeListener只做关键词匹配不够 |

### 4.2 核心差异化优势

```
CoC 7th Engine 的护城河不是单一功能，而是三重叠加：

   AI 守秘人叙事
      +
   CoC 7e 规则引擎 (非文本游戏，是真正的TRPG)
      +
   克苏鲁氛围沉浸 (SAN视觉反馈 + 氛围滤镜 + 暗黑主题)

→ 这三者叠加在"纯前端零服务器"的部署模式下，
  是目前市场上独一无二的产品定位。
```

### 4.3 具体优势拆解

1. **AI + 规则引擎 ≠ AI + 文字游戏**: ChatGPT可以讲克苏鲁故事但不会算伤害减免、不会掷疯狂发作表、不会追踪弹药。本引擎的差异化在于AI输出被Schema约束的Tool Call→规则引擎精确执行→状态写回→AI看到结果后继续叙事。这是一个**闭环系统**而非开放式文本生成。

2. **工具快照回滚**: `makeToolSnapshot()` / `restoreToolSnapshot()` 实现AI工具调用的事务性——AI"胡说"不会破坏游戏状态。这比普通跑团bot的"信任AI"要安全一个数量级。

3. **叙事监听器**: `narrativeListener()` 扫描AI纯文本输出，正则匹配自动触发线索发现、位置更新等系统操作。这意味着AI不需要显式调用所有工具，降低了对AI prompt engineering的依赖。

4. **中国TRPG生态位**: 中文CoC圈子活跃玩家估计10-30万量级，但缺KP是公认痛点。竞品Foundry/Roll20在中国几乎没有渗透率（英文界面+付费墙+服务器依赖），而跑团bot受制于QQ/DC平台限制。纯前端+全中文+免费是进入中国市场的三重门钥匙。

### 4.4 差异化风险

| 风险 | 说明 |
|------|------|
| **AI供应商锁定** | 硬编码DeepSeek，切换供应商需大量重写 |
| **单人体验天花板** | 多人联机缺失意味着无法覆盖目前最大的TRPG使用场景（朋友组队） |
| **大模型降智风险** | AI输出质量不稳定直接影响游戏体验——尤其是中小模型容易"忘记规则" |
| **Figma vs VTT 之争** | 如果Foundry VTT出官方AI GM插件，本引擎的地图/Token/联机短板将暴露 |

## 六、📋 战略优先级 — 「如果只剩1个Sprint」

### 6.1 背景假设
- 1 sprint = 2周 = 10个工作日 = ~80工程小时
- 当前状态: 18/18测试全部通过，bug_hunt 0/129，deployable
- 目标: 最大化「上线后用户留存率」×「开发风险可控」

### 6.2 三大候选方向

#### 🅰️ 方向A: 引擎拆分 (架构治理)

**任务**: 将coc.js(1665行)拆分为6个独立引擎模块（dice/attributes/combat/sanity/wound/mythos），同步修复分层违规。

**收益**:
- 降低未来的Bug修复成本 ~60%
- 每个模块可独立测试 (目前coc.js耦合严重，修改一处需全量回归)
- 为第三方开发者贡献代码打开大门

**风险**: 🔴 高——"引擎拆分破坏测试"在ROADMAP中被列为高风险项。需要每步跑全量测试，18/18门禁必须保持。

**ROI评价**: 对最终用户不可见，但对可持续开发至关重要。**技术债是复利的——越晚还利息越高。**

---

#### 🅱️ 方向B: 沉浸感升级 (克苏鲁性深化)

**任务**:
1. Web Audio 环境音效引擎 (心跳/低语/风声/深海) — 3天
2. SAN叙事反馈集成 — 低SAN时AI prompt注入偏执/幻觉指令 — 2天
3. 典籍阅读动画仪式感 — 1天
4. 气氛系统更多层级 (从4级扩展到7级) — 1天
5. 移动端适配冲刺 — 2天
6. 缓冲: 1天

**收益**:
- 直接提升用户"第一印象"留存率
- 视听沉浸感是与其他TRPG工具拉开差距的最直观方式
- 移动端适配解锁通勤/睡前等碎片化场景

**风险**: 🟡 中——音频兼容性(尤其是移动端Web Audio)需要大量测试。

**ROI评价**: 对demo/推广阶段收益最高。**用户不会因为架构优美而付费，但会因为"卧槽这个SAN下降屏幕在震"而截图发群。**

---

#### 🅲️ 方向C: 多人联机MVP (市场扩展)

**任务**: WebSocket/WebRTC基础联机 + 共享AI守秘人 + 小队投票检定

**收益**: 覆盖最大TRPG使用场景（2-5人组队），从单人工具扩展为社交平台。

**风险**: 🔴 极高——联机引入状态同步、冲突解决、网络重连等复杂问题。1个sprint只能做极简demo。

**ROI评价**: 战略方向正确但1个sprint不够。应该是V18-V20的长期目标，而非紧急冲刺。

---

### 6.3 创意总监最终裁决: **🅱️ 方向B — 沉浸感升级**

**理由**:

1. **差异化即生命线**: 这个引擎的最大优势不是"又一个VTT"而是"AI守秘人+克苏鲁沉浸"。方向B直接强化核心差异化，方向A/C要么不可见要么仓促。

2. **留存是第一指标**: CoC TRPG的平均session时长2-4小时。如果一个玩家第一次打开引擎、创建角色、跑了第一个遭遇(30分钟)、感受到了SAN下降的屏幕震动和低语音效——这个玩家留下继续跑完整个sessions的概率远高于"规则算得准但看着像Excel"的体验。

3. **架构债可以策略性拖延**: coc.js是1665行不是16650行。18/18测试通过意味着当前债不阻塞交付。V18 roadmap已规划拆分，应该在真正需要加新feature时才重构（YAGNI原则）。

4. **移动端是沉默的大多数**: 中文TRPG圈的手机用户远多于PC用户。2天移动端适配冲刺能解锁一个当前完全未覆盖的巨大用户群。

## 七、📝 最终建议

### 给CCGS管理层的三句话:

1. **这是一个好产品原型，不是一个好产品。** 引擎的创意核心(AI守秘人+规则引擎+氛围沉浸)锋利有力，但代码质量和多人联机的缺失意味着离"上线赚钱"还有1-2个季度的打磨。

2. **竞争窗口是12-18个月。** 现在中文TRPG市场没人做"AI守秘人+完整CoC引擎"这个组合。但大厂(字节/腾讯)一旦看到TRPG+AI的机会，用50人团队3个月就能做出竞品。速度是唯一的护城河。

3. **先做沉浸感，再做联机，最后重构。** 优先级错误是创业公司最常见的死因。用户不会因为你的代码干净而付费，但会因为"SAN归零的那一刻屏幕暗下来耳畔传来低语然后我尖叫了"而变成你的传教士。
